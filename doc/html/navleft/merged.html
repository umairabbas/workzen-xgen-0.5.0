site

Overview

[nav left] [nav top] [Merged one page docs]

XGen is a transformation engine. It provides a framework to load input data, and process that data into a different output formats. It has been used primarily as a source code generator using the Velocity template engine.

Loaders and templates have been build to generate:

  • JDBC persistence classes
  • POJO plain old java objects
  • Xml2Html conversions
  • Html documentation

News

July 2006

XGen has been refactored to remove Ant and Velocity dependencies. Ant functions only as an entry point to execute the transformation. Velocity is now a processor that is defined at runtime.

November 2004

XGen Documentation is updated and released.

April 2004

XGen is refactored and re-released under the workzen code tree. XGen now supports multiple template instances, allowing complex code builds. The persistence code has been seperated from individual classes, and support  for persistence of references and collections has been added.

July 2002

Initial release of XGen. XGen builds souce code that allows bean level persistence of objects.

Quick Start

Quick Start

  • Download XGen
  • Create an input file
  • Create an Ant build file
  • Run Ant

Scenario

Typically, I will explode the XGen distribution into the top level of my project tree, and create Ant build files for all source code needs. As the project changes, I can change the input schema, and regenerate my code as needed.

technical

Technical Information

XGen is a framework for managing transformations. Input data is loaded into an object model, and processed using a template engine. There are three components used in the transformation: loaders, processors and models.

Loaders

The loader is responsible for loading the object model. Source data used by the loader could be anything, a text file, and xml file, a database, a web service, etc.

Processors

A processor transforms the input data, and persists the data in some way.  A processor is typically a template engine. XGen provides a VelocityProcessor and templates to transform models. For more information on Velocity visit the Apache website.

Models

Object models are the key to the transformation. By building a model of the input data, it can be easily transformed. Anything can be modeled. XGen provides a number of models around which transformations can be performed.

And finally we need a way to execute the transformation. XGen provides an Ant task that invokes the loader, and causes the engine to process the data.

ant

XGen Ant Task

Ant is a Java build tool, used to manage software projects. It has been adapted here to configure the runtime components, and execute the process. You can read more about Ant from the Apache website.

The XGenTask has been "decoupled" from the transformation process. Earlier versions used the Velocity TexenTask, which is dependent on Velocity. We now only use Ant to configure the Loader and Processor, and invoke the execution.

Configuration

Ant is used by XGen as "property configurator". All properties defined in the task are passed to the Loader and the Processor as a standard Property class. XGen processors can be embedded in other programs without having a dependency on Ant.

Example

<?xml version="1.0"?>
<project name="xgen" default="build" basedir=".">

<!-- ====================================================================== -->
<!-- Example build -->
<!-- ====================================================================== -->

<property file="build.properties"/>
<property name="input.file" value="${basedir}/etc/input/blog.xml"/>
<property name="output.dir" value="${basedir}/work/blog"/>

<path id="classpath">
  <pathelement path="."/>
    <fileset dir="${basedir}">
    <include name="**/*.jar"/>
  </fileset> 
</path>

<target name="build">

<echo message="+-----------------------------------------------------+"/>
<echo message="| Generating project source! |"/>
<echo message="+-----------------------------------------------------+"/>

<taskdef name="xgen" classname="workzen.xgen.ant.XGenTask">
<classpath refid="classpath"/>
</taskdef>

<!-- pojo --> 
<xgen
  loader="workzen.xgen.loader.JavaModelLoader"
  xmlFilePath="${input.file}"
  processor="workzen.xgen.engine.VelocityProcessor"
  templateBasePath="${basedir}/src/workzen/xgen"
  templatePath="templates/pojo"
  template="control.vm"
  outputPath="${output.dir}"
  outputFile="pojo.out"
>
</xgen> 

</target>

</project>
This example loads the Java Package model from an xml input file, and uses the Velocity template engine to process it.

loaders

Loaders

All loaders implement the ILoader interface. Loaders are self configuring. If a required property is not defined, and exception is thrown.

public interface ILoader {

    public void configure(Map map) throws XGenException;

    public Object loadModel() throws XGenException;

}

Configuration

Loaders uses properties defined by the Ant task to configure itself.

loader="workzen.xgen.loader.JavaModelLoader"
xmlFilePath="${input.file}"

The loader classname must be defined in order for XGen to instantiate the class. All properties defined in the task are passed to the loader as a Map. The JavaModelLoader uses an xml file for input.

processors

Processors

All processors implement the IProcessor interface. Processors are self configuring. If a required property is not defined, and exception is thrown.

public interface IProcessor {

    public void configure(Map map) throws XGenException;

    public void process(Object model) throws XGenException;

}

VelocityProcessor

The VelocityProcessor uses properties defined by the Ant task to configure itself.

processor="workzen.xgen.engine.VelocityProcessor"
 templateBasePath="${basedir}/src/workzen/xgen"
 templatePath="templates/persistence/manager"
 template="control_table.vm"
 outputPath="${output.dir}"
 outputFile="table.out"

The processor classname must be defined in order for XGen to instantiate the processor. All properties defined in the task are passed to the processor and will be added to the VelocityContext. In addition to configuring the engine, properties can be extracted from anywhere within a template.

Two path properties are defined in the task.

  • templateBasePath is used by Velocity to search the filesystem for the template
  • templatePath is used by Velocity to search the classPath and jar files for the template.

The order of the search is "file,classpath". This allows the developer to customize templates as needed from the filesystem.

Java Package Model

The workzen.xgen.model.Package class models a standard Java object oriented model. It provides accessors to all OO elements: classes, attributes, contained objects and references.

Input

A java model could be created from a number of sources; manually, using introspection, using an rdbms schema, or an xml file.

xml schema

JavaModel XML Schema

Package

The package tag defines the Java Package.

 <package name="workzen.student.model">
attribute value description
name String fully qualified package name

Class

The class tag defines the Java Class and the RDBMS table.

 <class name="Student" 
   tablename="a_student" 
   handle-concurrency="false" 
   handle-autoincrement="true" 
   author="java developer" 
   description="Student model"
   superclass="Person"
  >
attribute value description
tablename String RDBMS tablename
handle-concurrency boolean Use XGen to handle concurrency via "tcn" field in table
handle-autoincrement boolean Use XGen to handle autoincrement via "select sum" on primary key column
author String Used in class comments
description String Used in class comments
superclass boolean The superclass name for the "extends" parameter
imports CSV Comma seperated string of package imports

Field

The field tag defines the Java class attribute.

Note: All of the column attributes can be defined within the field tag. This is part of the "legacy" schema, but should be considered deprecated.

<field name="id" type="long" primitive="true">
attribute value description
name String Java class attribute
type String XGen type (see types)
primitive boolean Is the type a primitive

Column

The column tag is defines the RDBMS table column.

<column name="class_id" null="false" primary-key="true" autoincrement="false" />
attribute value description
name String table column name
null boolean is the column nullable
primary-key boolean is the column a primary key
foreign-key boolean is the column a foreign key
transaction-key boolean is the column a transaction key
autoincrement boolean is the column an autoincrement key (required?)
size int number of characters for a String/Varchar field
precision int numeric precision for float field
scale int numeric scale for a float field

Reference

The reference tag defines the Java reference.

<reference name="schedule" class="Schedule" key="scheduleId" />
attribute value description
name String Java reference attribute name
class String The classname of the reference
key String The local "id" field name that is used as a foreign-key

Collection

The reference tag defines the Java reference.

<collection name="scheduleItems" class="ScheduleItem" type="Vector" reverse-key="id" />
attribute value description
name String Java attribute name for the collection
class String The classname of the classes in the collection
type String The Java type of the collection
reverse-key String The "id" field name of the contained class

JDBC Table Class

The generated JDBC classes can persist any Java class with standard accessors, and "id" fields. It can also persist contained classes and references. See the examples.

Goal

The goal of this persistence framework is to produce human readable JDBC code that can be managed at a developer level.  Many frameworks exist to persist Java classes via xml schema definitions. For example:

  • Apache Torque
  • Apache OJB

Because persistence behavior is complex, I have found it easier to control the behavior of contained relationships via code rather than complex xml schema manipulation. This project can generate 95% of your JDBC code.

Table Level Persistence

All persistable classes have a corresponding SQL table. Table columns relate directly to Java class "fields".

Manager Classes

Each persistable class also has a "manager" class that manages the contained relationships. You can control, through code comments, which contained relationships get persisted, or deleted.

ID Fields

Each class needing persistence must include a numeric "id" field. This id field corresponds to the primary key field of the table row. In addition, each reference needing persistence, must also have an "id" field. This will be populated by the foreign-key. This is similar to how OJB and Torque persist their classes.
